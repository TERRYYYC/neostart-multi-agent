# 03 A2A 调用

## A. 观察事实
- 主消息流中存在多只猫的连续回复，头像与卡片颜色区分不同角色。
- 某些消息包含折叠的“工具调用”区域，并显示调用次数与工具结果。
- 可见一条由布偶猫输出的大块结构化内容，前面紧接着缅因猫的工具调用消息。
- 右侧状态栏仍同步展示各猫 session 使用情况。
- `DEMO.md` 明确说明支持 `A @ B`、链式 `A @ B @ C` 和完整上下文传递。

## B. 推断
1. 推断: A2A 调用在公开聊天流中表现为“结果消息入流”，而中间执行过程只部分可见。
   - 证据: 看到工具调用折叠块和最终回复，但看不到完整内部推理。
   - Confidence: high
   - Validation needed: 确认发起猫与被调用猫之间是否还有私有消息通道。
2. 推断: A2A 是消息级调度，而不是简单共享同一个全局上下文。
   - 证据: `DEMO.md` 强调“自动路由到缅因猫”“形成对话链”“完整上下文传递”。
   - Confidence: high
   - Validation needed: 确认上下文传递是全文注入、摘要注入，还是消息引用。
3. 推断: A2A 在产品定位上是核心能力，不是增强功能。
   - 证据: 文档把它放在“多猫协作”主段落，并直接展示真实界面结果。
   - Confidence: high
   - Validation needed: 确认若关闭 A2A，系统是否仍有足够差异化价值。

## C. 待确认问题
- 发起猫调用其他猫时，用户是否总能看到“这是 A2A 触发”的系统提示？
- 链式调用的深度是否受系统配置限制？
- 被调用猫能否再次调用回原猫，从而形成循环？
- 失败调用在 UI 中如何呈现？

## D. v1 范围建议
- v1 先支持单跳 A2A 与有限深度链式调用。
- 先不做复杂仲裁、多猫投票、自动策略选猫。
- 上下文传递可先用“最近消息摘要 + 明确任务指令”，不必一开始实现全量长上下文搬运。

## E. 架构建议
- Presentation layer: A2A 最少需要一种可辨识标记，告诉用户“这条消息是被哪只猫转交触发的”。
- Interaction layer: 引入 `AgentInvocation` 概念，允许一条公开消息派生多个子调用。
- Capability layer: 需要一个编排器处理目标猫解析、权限校验、上下文打包、结果回流。
- 跨层推断: 如果没有 invocation 树，后续无法解释 session 消耗、状态栏更新和 A2A 输出隔离之间的关系。
