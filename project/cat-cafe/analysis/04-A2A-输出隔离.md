# 04 A2A 输出隔离

## A. 观察事实
- 当前没有本地截图文件可供核对。
- `DEMO.md` 明确给出的描述是：
  - 被调用猫的 CLI thinking、工具调用日志是私有的。
  - 只有主动 `post_message` 发送的内容才会出现在聊天室。
  - 技术划分为 CLI stdout/stderr 内部通道与 MCP callback 公开通道。

## B. 推断
1. 推断: 系统把“执行日志”与“公开消息”当成两个不同消息总线。
   - 证据: 文档直接区分内部通道与公开通道。
   - Confidence: high
   - Validation needed: 需要界面或日志样例验证这两个通道的落点。
2. 推断: A2A 输出隔离不是隐私附加项，而是防止多代理互相污染聊天室的必要机制。
   - 证据: 如果没有隔离，多猫工具日志会淹没主聊天流。
   - Confidence: high
   - Validation needed: 需要确认哪些元信息仍允许公开，比如工具调用次数或执行中状态。
3. 推断: 产品后端需要明确定义事件可见性级别，如 `public / private / system-summary`。
   - 证据: 单靠前端隐藏无法满足“只有猫自己能看到”的语义。
   - Confidence: high
   - Validation needed: 需确认用户是否需要管理员视角查看全部内部日志。

## C. 待确认问题
- 用户是否完全看不到内部日志，还是能在专门面板查看摘要？
- 内部工具失败是否会转换成公开错误消息？
- A2A 私有日志是否会参与长期记忆或 session handoff 摘要？
- 是否存在“开发模式”，允许暂时公开更多执行信息？

## D. v1 范围建议
- 即使没有截图，v1 也应把“公开消息”和“内部执行日志”彻底分离，这属于基础架构而非装饰功能。
- 前端可以先只展示公开消息和少量系统摘要，不开放内部日志查看器。
- 如果后期需要调试模式，再增加“开发者可见”的 meta 面板。

## E. 架构建议
- Presentation layer: 聊天主流只承载 `public message` 与极少量 `system summary`。
- Interaction layer: 被调用猫完成后主动提交结果；未提交的思考与工具事件不进入聊天区。
- Capability layer: 事件模型建议带 `visibility` 字段，至少支持 `public`、`private-agent`、`system`.
- 跨层推断: 这是 `03 A2A 调用` 和 `09 CLI Meta` 能同时成立的前提，否则公开聊天流会与调试视图混杂。
