# Reverse Engineering Brief

## 1. Product Goal
- Target product: Cat Café，一个多 AI 代理协作聊天与任务执行系统。
- Why replicate it: 其核心价值不在单聊天界面，而在多代理协作、可见状态、配置治理、上下文连续性这四类能力的组合。
- Intended outcome: 现阶段应以 MVP 架构定义为目标，不是像素级 UI 复刻。

## 2. Replication Target
- Presentation layer: 复刻三栏工作台、中心对话流、右侧状态栏、Hub 配置中心这几个一级界面结构。
- Interaction layer: 复刻 `@` 召唤、多猫并行回复、A2A 调用、状态变化、session 交接这几条关键工作流。
- Capability layer: 复刻多代理调度、权限与配置管理、上下文预算控制、A2A 消息路由。

## 3. Known Screens / Known Flows
- Screen 00: 新建对话弹窗，可选默认猫猫与项目目录。
- Screen 01: 主聊天界面右侧状态栏，展示猫状态、消息统计、session chain。
- Screen 02: `Cat Café Hub` 配置中心，含按猫配置与系统配置、命令速查、环境与文件、语音、通知。
- Screen 03: A2A 协作后的消息流与工具调用折叠块。
- Screen 04: 无本地截图，仅有 DEMO 文本描述，主题为 A2A 输出隔离。
- Screen 05: 右侧 `Session Chain` 展示多 session、活跃/封存状态、预算占用与阈值触发。

## 4. Core User Journey
- Entry: 用户进入某个 Thread，或通过“新对话”创建新 Thread。
- Main action: 用户通过 `@猫猫` 指定一个或多个代理处理问题。
- Feedback: 中央消息流出现多猫回复，右侧状态栏持续反馈工作状态和 session 使用情况。
- Completion: 用户得到公开回复；内部推理、工具日志与 session 交接由系统处理。

## 5. Candidate Data Entities
- Thread
  - Purpose: 承载一个话题级工作空间。
  - Key fields: `id` `title` `workspacePath` `pinned` `archivedAt` `lastActiveAt`
- Message
  - Purpose: 聊天区公开消息。
  - Key fields: `id` `threadId` `role` `authorType` `authorId` `contentBlocks` `mentions` `createdAt`
- AgentProfile
  - Purpose: 每只猫的身份与默认行为配置。
  - Key fields: `id` `name` `provider` `model` `persona` `contextBudget` `toolPolicy`
- AgentSession
  - Purpose: 某只猫在某个 Thread 中的单段运行上下文。
  - Key fields: `id` `threadId` `agentId` `ordinal` `status` `tokenUsage` `cacheUsage` `startedAt` `sealedAt`
- AgentInvocation
  - Purpose: 一次用户召唤或 A2A 调用的执行记录。
  - Key fields: `id` `threadId` `sourceMessageId` `targetAgentId` `visibility` `state` `startedAt` `finishedAt`
- ToolEvent
  - Purpose: 工具调用与内部执行日志。
  - Key fields: `id` `invocationId` `toolName` `summary` `visibility` `startedAt` `finishedAt`
- SessionHandoff
  - Purpose: session 交接链条与摘要衔接。
  - Key fields: `id` `fromSessionId` `toSessionId` `reason` `triggerThreshold` `summaryRef`
- WorkspaceBinding
  - Purpose: Thread 与项目目录、环境配置的绑定。
  - Key fields: `id` `threadId` `path` `configSource`

## 6. Architecture Draft
- Frontend: 三栏 SPA。左侧 Thread 导航，中部消息与交互，右侧状态/统计/chain；另有全局 Hub 配置中心。
- Backend: 需要一个代理编排层，负责消息路由、模型调用、工具调用、A2A 上下文转发、session handoff。
- Storage: 至少需要持久化 Thread、公开消息、代理配置、session 元数据、handoff 摘要；内部日志可分冷热存储。
- External services: LLM provider、语音能力、可选长期记忆服务、可选通知服务。

## 7. v1 Scope
- In scope:
  - 单用户、多代理协作 Thread
  - `@` 召唤与多猫并行回复
  - 基础状态栏
  - 每猫基础配置
  - A2A 路由
  - Session Chain 的最小闭环
- Out of scope:
  - 语音输入输出
  - 富文本复杂块
  - 推送通知
  - 长期记忆深度集成
  - 游戏化玩法

## 8. Risks / Unknowns
- Risk 1: `04 A2A 输出隔离` 缺少截图，当前只能依据 DEMO 文本推断。
- Risk 2: `02 配置中心` 暗示系统已存在复杂运行时治理，MVP 若全量复刻会过早扩大后端范围。
- Risk 3: `05 Session Chain` 是否真需要自动新开 session，还是可以先用“摘要压缩同 session”替代，需尽早定策略。
- Risk 4: 截图来源于 demo 录屏，部分字段可能是研发态信息，不一定都该进入产品态。

## 追加修正（基于 08）
- `Thread` 现在可以确认是一等资源，而不是左侧聊天列表的轻量 UI 包装。
- 右侧状态栏不只是展示状态，还承担 `当前调用` 与 `审计日志入口` 的追踪职责。
- `invocation` 已在界面中直接露出，这进一步说明 `AgentInvocation` 应进入核心数据模型，而不是内部实现细节。

## 架构问题的当前回答
1. 目标更接近“能力复刻 + 交互复刻”，不是单纯 UI clone。
2. v1 必需能力是多猫召唤、A2A 路由、状态反馈、基础配置、session 连续性。
3. 昂贵但非必需的是完整 Hub 全配置、长期记忆、通知、语音、命令体系全量复刻。
4. 核心实体见上文 `Thread / Message / AgentProfile / AgentSession / AgentInvocation / ToolEvent / SessionHandoff / WorkspaceBinding`。
5. 客户端持有临时 UI 状态与订阅状态；服务端持有执行状态、session、日志、配置真相源。
6. 需要持久化；至少是会话型持久化，建议消息与 session 元数据入库，工具细节可分层保存。
7. 流式行为属于核心体验的一部分，尤其是状态变化与多猫并行响应。
8. 需要轻量任务编排；是否需要完整后台队列取决于 tool call 与 handoff 是否异步化。
9. 当前更像长期产品的第一阶段，不应按 throwaway prototype 设计。
10. 必须保持灵活的决策包括：session 策略、内部日志公开边界、长期记忆是否内建、配置中心深度。

## 推荐路径
- 推荐方案: 先做“单 Thread、多代理、公开消息 + 私有执行日志 + 右侧状态栏 + 最小 handoff”的垂直切片。
- 备选方案: 如果 handoff 成本过高，v1 先只展示预算进度和手动压缩，不自动创建新 session。
