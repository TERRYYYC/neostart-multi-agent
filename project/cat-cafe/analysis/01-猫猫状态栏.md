# 01 猫猫状态栏

## A. 观察事实
- 右侧存在固定“状态栏”区域。
- 状态栏至少包含三个模块：`猫猫状态`、`消息统计`、`Session Chain`。
- `猫猫状态` 模块当前显示“等待调用...”。
- `消息统计` 展示总数、猫猫消息、系统消息、Evidence、Follow-up 等计数。
- `Session Chain` 中每个卡片显示 session 编号、关联猫猫、启动时间、token/cache 数据、百分比进度和状态标签。
- 主聊天区右边还有一列与消息对齐的彩色圆点，疑似用颜色关联不同猫或 session。

## B. 推断
1. 推断: 状态栏是全局运行态面板，不只反映当前一条消息。
   - 证据: 同时出现消息统计与多 session 卡片，而不是单消息状态。
   - Confidence: high
   - Validation needed: 确认切换 Thread 后是否整体刷新。
2. 推断: `消息统计` 中的 Evidence / Follow-up 是系统内部消息类型，而非普通文本标签。
   - 证据: 它们与“总数/系统消息”并列计数。
   - Confidence: medium
   - Validation needed: 确认这些类型是否会单独渲染成特定 block。
3. 推断: Session 卡片的百分比是 context 使用率，彩色标签与猫猫身份绑定。
   - 证据: `05` 文本说明 context 预算与 handoff；卡片颜色与猫猫品牌色一致。
   - Confidence: high
   - Validation needed: 确认百分比是否由 token、cache 或综合权重计算。

## C. 待确认问题
- “思考中 / 工具调用 / 回复中 / 空闲”这些状态是按猫维度还是按 invocation 维度展示？
- 状态栏是否支持点击进入更详细的 meta 视图？
- 彩色圆点列是时间线辅助还是 session 轨迹？
- 统计口径是当前 Thread 还是当前可见消息窗口？

## D. v1 范围建议
- v1 先保留三块：猫猫状态、基础消息统计、session 列表。
- 可以不做 Evidence / Follow-up 的复杂语义，只保留类型计数占位。
- 若后端压力较大，先用轮询或 SSE 更新状态，不必一开始就做双向实时同步。

## E. 架构建议
- Presentation layer: 右栏必须独立于消息流，避免随着聊天滚动失去运行态可见性。
- Interaction layer: 状态栏订阅 `AgentInvocation` 和 `AgentSession` 的实时事件流。
- Capability layer: 服务端应把“公开消息”和“运行元数据”分流；右栏消费的是后者的聚合结果。
- 跨层推断: 既然状态栏承担全局反馈职责，前端状态树应把 Thread 内容与运行态拆开，否则多猫并发时会出现局部刷新耦合问题。

## F. 下一步决策
- 最有价值的下一步输入是：补一段状态变化序列截图，尤其是“思考中 -> 工具调用 -> 回复中 -> 空闲”的连续过程。
