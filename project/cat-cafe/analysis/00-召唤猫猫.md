# 00 召唤猫猫

## A. 观察事实
- 画面显示点击左上角 `+新对话` 后，中央出现“新建对话”弹窗。
- 弹窗包含“默认猫猫（可选）”区域，并按 Anthropic / OpenAI / Google 分组。
- 每组内可以选择一个或多个猫猫选项，截图中可见布偶猫、缅因猫、暹罗猫及其模型变体。
- 弹窗包含“选择项目目录”区域，列出多个目录项，底部有“浏览其他目录”。
- 背景页面仍是三栏主界面，说明该操作是在当前工作台上方以模态方式完成。

## B. 推断
1. 推断: 新对话创建时同时完成“Thread 初始化 + 默认参与代理绑定 + 工作区绑定”。
   - 证据: 弹窗同时要求默认猫猫和项目目录。
   - Confidence: high
   - Validation needed: 确认如果不选目录，Thread 是否仍可创建。
2. 推断: 默认猫猫支持多选，且会影响后续消息默认路由或可见身份集合。
   - 证据: 同一弹窗里存在多个可点击猫选项，DEMO 文本说明支持同时 `@` 多只猫。
   - Confidence: medium
   - Validation needed: 确认“默认猫猫”与消息里显式 `@` 的优先级关系。
3. 推断: 项目目录是该系统的一等公民，不只是附件来源。
   - 证据: 目录选择在创建 Thread 时出现，而非埋在高级设置。
   - Confidence: high
   - Validation needed: 确认一个 Thread 是否可切换或绑定多个目录。

## C. 待确认问题
- 默认猫猫是“新 Thread 的默认参与者”还是“首次发言自动召唤对象”？
- 创建后是否允许修改工作区目录？
- 一个 Thread 是否支持多 workspace 绑定？
- 不选默认猫猫时，系统是否允许仅靠后续 `@` 临时召唤？

## D. v1 范围建议
- 先做最小创建流：标题自动生成、单目录绑定、默认猫猫多选。
- 不在 v1 处理复杂目录权限、最近目录历史、跨目录索引。
- 若想压缩复杂度，默认猫猫可先只支持三只基础身份，不暴露模型变体。

## E. 架构建议
- Presentation layer: 保持模态式创建流，不建议拆去独立页面，因为它明显依附现有工作台上下文。
- Interaction layer: 创建动作产出一个 `Thread` 和若干 `ThreadAgentBinding`。
- Capability layer: 后端需要校验工作区路径、记录默认代理集合，并为后续 A2A / tool 权限做初始化。
- 跨层推断: “目录绑定在创建时完成”意味着未来很多工具权限和上下文装配应从 `Thread` 级别继承，而不是每条消息重复选择。

## F. 下一步决策
- 最有价值的下一步输入是：补一张“创建完成后首次发言”的截图，用来判断默认猫猫到底如何参与首轮执行。
